# Readme Redis

## Описание сервиса

Redis в проекте используется как общее in-memory хранилище счетчиков для rate limiting (WAF-слой).

Текущая инициализация:
- клиент создается в `backend/src/Cache/RedisFactory.php`;
- настройки берутся из переменных окружения:
  - `REDIS_HOST` (по умолчанию `127.0.0.1`);
  - `REDIS_PORT` (по умолчанию `6379`);
- драйвер: `predis`.

## Где сейчас используются лимиты и WAF-счетчики

RateLimitMiddleware с Redis подключен в `backend/public/index.php` на следующих endpoint:

- `POST /auth/login` - лимит `20` запросов за `60` секунд;
- `POST /auth/register` - лимит `10` запросов за `60` секунд;
- `POST /notes` - лимит `20` запросов за `60` секунд.

## Принцип работы

Реализация находится в `backend/src/Http/Middleware/RateLimitMiddleware.php`.

Алгоритм:
- формируется ключ вида `rate:{METHOD}:{PATH}:{IP}`;
- выполняется `INCR` в Redis;
- если значение стало `1`, на ключ ставится `EXPIRE` с длиной окна;
- при каждом запросе читается `TTL`;
- если счетчик превысил лимит, возвращается `429 Too Many Requests` и `Retry-After`;
- иначе запрос пропускается дальше по цепочке middleware.

Диагностические заголовки в успешном ответе:
- `X-RateLimit-Count`;
- `X-RateLimit-TTL`.

Поведение при недоступности Redis:
- fail-open (доступность выше жесткой блокировки);
- запрос не блокируется;
- выставляется `X-RateLimit-Bypass: 1`;
- в лог пишется сообщение о недоступности Redis.

## Роль Redis в WAF

Redis выполняет роль быстрого и общего хранилища счетчиков между процессами `php-fpm` и инстансами приложения.

Это дает:
- консистентный лимит для всех воркеров;
- низкую латентность операций инкремента;
- простую реализацию TTL для окон лимитирования;
- предсказуемое поведение под параллельной нагрузкой.

Без Redis in-memory счетчики внутри процесса работают некорректно в multi-process окружении и не дают общего лимита.

## Аналитика принятия решения

### Контекст
- Рассматривалась замена Redis на Postgres (Postgres 16) для унификации стека и упрощения сопровождения.
- Цель: понять, можно ли убрать отдельный сервис Redis без потери надежности и управляемости WAF.

### Возможности Postgres для похожего функционала
- таблица счетчиков + `INSERT ... ON CONFLICT ... DO UPDATE` (`UPSERT`);
- скользящее окно через таблицу событий с `COUNT` по интервалу;
- бакеты времени (секунда/минута) с агрегированием;
- очистка старых записей через cron/планировщик.

### Плюсы подхода на Postgres
- один инфраструктурный инструмент (проще деплой и базовое сопровождение);
- единый источник данных и транзакционность;
- удобно совмещать с аудитом в одной БД.

### Минусы подхода на Postgres
- горячая write-нагрузка WAF конкурирует с бизнес-нагрузкой БД;
- выше риск контеншена и роста latency на критичных endpoint;
- дополнительная операционная работа по очистке старых счетчиков и контролю блоата;
- Redis лучше подходит для частых атомарных инкрементов с TTL.

### Принятое решение
- Оставить текущую реализацию на Redis.
- Основание: Redis лучше соответствует профилю WAF-счетчиков (быстрые инкременты, TTL, общий state между воркерами), при этом уже внедрен и работает в проекте.
- Решение повторно пересмотреть при изменении профиля нагрузки или инфраструктурных ограничениях.
