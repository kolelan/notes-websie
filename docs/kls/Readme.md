## KLS: универсальная система классификаторов

В папке `docs/kls` собраны SQL-скрипты DDL для подсистемы классификаторов в схеме `kls`.

Цель подсистемы:
- хранить классификаторы (`qual`) и их разделы (`kls`);
- поддерживать иерархию разделов через `ltree` (`kls_rubrika`);
- предоставлять представления (`view`) и вспомогательные функции для поиска, выгрузки и построения дерева.

## Структура

- `sql/*_ddl.sql` - DDL-скрипты таблиц, view, materialized view, функций.
- основные сущности:
  - `kls.qual` - классификатор;
  - `kls.kls` - разделы классификатора;
  - `kls.kls_kls` - связи "родитель-дочерний";
  - дополнительные специализированные таблицы (`kls_okato`, `kls_oksm`, `kls_okei` и др.).

## Обновление: поля `tags` и `tag`

Добавлено новое необязательное поле:
- `kls.kls.tags hstore NULL`
- `kls.qual.tag hstore NULL`

Назначение:
- хранение произвольных тегов раздела в формате `hstore` (ключ/значение);
- быстрый способ расширять метаданные без изменения структуры таблицы.

Важно:
- тип `hstore` требует наличия расширения PostgreSQL `hstore` в БД.

Изменения внесены в:
- `sql/kls_ddl.sql` - добавлена колонка `tags`;
- `sql/qual_ddl.sql` - добавлена колонка `tag`;
- `sql/v_kls_ddl.sql` - `tags` добавлен в состав view `v_kls`;
- `sql/v_kls_atd_ddl.sql` - `tags` добавлен в состав view `v_kls_atd`.
- `sql/v_qual_ddl.sql` - `tag` добавлен в состав view `v_qual`;
- `sql/v_kls_ddl.sql`, `sql/v_kls_atd_ddl.sql`, `sql/v_kls_kls_ddl.sql`, `sql/v_qual_kls_ddl.sql`, `sql/v_kls_subject_okato_ddl.sql` - добавлены поля `qual_tag`/`qual_tag_parent`/`qual_tag_child`.

## Снятие внешних зависимостей

Для повышения универсальности из SQL-объектов убраны внешние зависимости на другие схемы:
- удалены профильные ограничения доступа из `v_qual` и `v_kls` (ранее `sec.*` + `spo.current_profile_id()`);
- удалена зависимость `opt.kls_way` из `v_kls_atd` (поле `way` теперь заполняется `NULL::text`);
- из процедурных скриптов убраны вызовы `service.*` и `spo.*`:
  - `r_kls_tree_data_check_ddl.sql` использует нативный `REFRESH MATERIALIZED VIEW`;
  - `r_kls_download_ddl.sql` переписан на автономную версию без внешних функций.

## Порядок применения

Рекомендуемый порядок (минимально):
1. таблицы (`qual`, `kls`, служебные связи);
2. view и materialized view;
3. функции и процедуры;
4. grants/comment (если отделены).

При обновлении существующей БД:
- сначала применить `ALTER TABLE` для добавления `tags`;
- затем пересоздать/обновить зависимые view (`v_kls`, `v_kls_atd`).

Готовый миграционный скрипт для существующей БД:
- `sql/m_20260407_add_hstore_tags.sql`
  - создает `hstore` extension (если не установлен),
  - добавляет `kls.qual.tag` и `kls.kls.tags`,
  - пересоздает затронутые view в актуальной версии.
